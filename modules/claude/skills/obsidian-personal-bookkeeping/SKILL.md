# AkiraのObsidian Bookkeepingガイド

## 領域の構造

```
Bookkeeping/
├── YYYY-MM.md       月次集計（dataviewjsで自動生成、手で書かない）
├── entries/         個別仕訳エントリ
│   └── YYYYMMDD-<摘要>.md
└── 折半判定.csv     折半対象の判定ルール（参考）
```

会計方式は**複式簿記**。1件の取引につき1ファイル、`debit`（借方）と `credit`（貸方）のリストを持つ。

## 仕訳エントリのファイル形式

ファイル名：`Bookkeeping/entries/YYYYMMDD-<摘要>.md`

```yaml
---
ID: 202603142305
created: 2026-03-14 23:05
title: <摘要>
tags: [仕訳/2026/01/06, 費目/費用/食費, 費目/負債/Paypayカード]
date: 2026-01-06
debit:
- category: 費用/食費
  amount: 537
credit:
- category: 負債/Paypayカード
  amount: 537
---

## メモ
```

## ファイル名と各日付フィールドの意味

3つの「日付」が登場するが、すべて意味が違うので混同しない：

| 場所 | 形式 | 意味 |
|---|---|---|
| ファイル名先頭 `YYYYMMDD` | 8桁 | **仕訳発生日**（実際に取引が起きた日） |
| `date:` フィールド | `YYYY-MM-DD` | **仕訳発生日**（ファイル名と一致） |
| `tags:` の `仕訳/YYYY/MM/DD` | スラッシュ | **仕訳発生日**（ファイル名と一致、表記違い） |
| `ID:` | `YYYYMMDDhhmm` | **入力時刻**（記帳した瞬間、後追い入力なら別の日になる） |
| `created:` | `YYYY-MM-DD HH:mm` | **入力時刻**（IDと同時刻） |

**実例**：2026-01-06の取引を2026-03-14に後から記帳した場合
- ファイル名：`20260106-セブンーイレブン.md`
- `date: 2026-01-06`
- `tags: [仕訳/2026/01/06, ...]`
- `ID: 202603142305`
- `created: 2026-03-14 23:05`

ユーザが「昨日の支出を記帳して」と言った場合は `date` は昨日、`ID`/`created` は今この瞬間。

## debit / credit の構造

**YAMLのリスト形式を厳密に守る**。dataviewjsクエリが `page.debit[i].category` のように各要素にアクセスするため、形式が崩れると月次集計が壊れる。

```yaml
debit:
- category: 費用/食費
  amount: 537
credit:
- category: 負債/Paypayカード
  amount: 537
```

**1つの取引で複数の費目に分けることも可能**：
```yaml
debit:
- category: 費用/食費
  amount: 1200
- category: 費用/日用品
  amount: 800
credit:
- category: 負債/Paypayカード
  amount: 2000
```

**借方合計 = 貸方合計** が成立すること。月次集計ページがこの一致を検証して、ずれていたら警告を出す。

## 費目（category）の階層

`category` は `<大分類>/<中分類>[/<小分類>]` の階層構造。典型的なもの：

### 借方によく出るもの（支出）
- `費用/食費`
- `費用/光熱費`
- `費用/通信費`
- `費用/交際費`
- `費用/日用品`
- `費用/交通費`
- `費用/医療費`
- `費用/書籍`
- `費用/サブスクリプション`

### 貸方によく出るもの（支払い手段）
- `負債/Paypayカード`
- `負債/ゴールドポイントカード`
- `負債/<その他カードブランド>`
- `資産/現金`
- `資産/<銀行口座名>`

### 収入の仕訳（借方=資産、貸方=収入）
- 借方：`資産/<銀行口座>`
- 貸方：`収入/給与` `収入/還付金` など

**新しい費目を導入する前に、既存entriesから抽出されたカテゴリ一覧から選ぶ慣習**。Templater (`BK-ENTRY.md`) は実際に既存ファイルを舐めて重複なくサジェストするが、Claudeが新規作成するときも同じ慣習を尊重する：費目を増やす前に「既存のどれかで代用できないか」を1度考える。本当に必要なら新規導入してよいが、その場合ユーザに「新しい費目『X』を作るが問題ないか」と1問確認するのが安全。

## tagsの構成

```yaml
tags: [仕訳/2026/01/06, 費目/費用/食費, 費目/負債/Paypayカード]
```

- 1つ目：`仕訳/<仕訳発生日のYYYY/MM/DD>`
- 2つ目以降：その仕訳で使われた**すべての費目**を `費目/<階層>` の形で列挙（debitとcreditを統合し重複を除く）

### 折半タグ

夫婦・パートナー間で折半する費用には `仕訳/折半` を追加する：

```yaml
tags: [仕訳/2026/01/06, 仕訳/折半, 費目/費用/食費, 費目/負債/Paypayカード]
```

月次集計ページの「折半」セクションはこのタグを持つ仕訳を自動的に集計して、合計の半額を表示する。何が折半対象かは `折半判定.csv` に判定ルールが書かれている。

## 月次集計ページの取り扱い

`Bookkeeping/2026-01.md` のような月次ファイルは、**dataviewjsで自動表示されるダッシュボード**。

- **新規作成する場合**は既存月（例：`2026-01.md`）をそのままコピーして、上部の `const targetYear = 2026;` と `const targetMonth = 1;` の値だけ書き換える
- **手で表を書かない**：すべてentriesから動的に集計される
- 出納セクションが全仕訳、折半セクションが `仕訳/折半` タグ付き仕訳を表示

## 作成ワークフロー

ユーザが「○○を××円使った」と言ってきた場合：

1. **摘要を決める**：店名・取引先名がベスト（例：「セブン-イレブン」「Amazon」「ヨークベニマル」）。短く具体的に
2. **仕訳発生日を確定**：明示があればそれを使う、なければ今日。ユーザに確認する場合は1問だけ
3. **借方を決める**：支出なら `費用/<カテゴリ>`、収入を受け取ったなら `資産/<口座>`
4. **貸方を決める**：支払い手段（カード／現金／口座）。ユーザがカード明細を貼っているなら `負債/<カード名>`、現金支払いなら `資産/現金`
5. **金額を確認**：借方=貸方が成立するか
6. **折半判定**：共通費用（食費・光熱費など、ユーザのルールによる）なら `仕訳/折半` タグを追加
7. **ファイル名生成**：`YYYYMMDD-<摘要>.md`
8. **`obsidian_append_content` で `Bookkeeping/entries/` に新規作成**

## 集計・参照パターン

| 求められるもの | 推奨される操作 |
|---|---|
| 今月の支出は？ | 当月の `Bookkeeping/YYYY-MM.md` を取得（dataviewjsが集計） |
| 特定費目の集計 | `obsidian_complex_search` で `費目/<カテゴリ>` タグでフィルタ、または entries/ を舐めて `debit`/`credit` を集計 |
| 特定店舗の履歴 | `obsidian_simple_search` で店名を全文検索 |
| 折半合計 | 月次ファイルの折半セクションを見る、または `仕訳/折半` タグでフィルタ |

## よくある落とし穴

- **拡張子の二重 `.md.md`**：`Bookkeeping/entries/` には過去に重複拡張子のファイルが大量に存在する（過去のテンプレートのバグ跡）。**新規作成時は単一の `.md` で書く**。既存の `.md.md` を発見しても勝手にrename/削除しない（ユーザに確認してから対応する）
- **借方≠貸方**：YAMLでは何でも書けてしまうので、Claudeが計算を間違えると気付かれにくい。書き込む前に必ず合計を検算する
- **費目の表記揺れ**：`費用/食費` と `費用/食事` のような揺れが入ると集計が分裂する。既存entriesと表記を揃える
- **`date` の `-` と `仕訳/` タグの `/` を取り違える**：`date: 2026-01-06` （ハイフン）と `tags: [仕訳/2026/01/06, ...]` （スラッシュ）。同じ日を別記法で書く点に注意
- **tags のインライン記法**：`tags: [...]` のインライン記法で書く（Output系ノートと同じ慣例）

## 作成前の最終チェック

- [ ] ファイル名は `Bookkeeping/entries/YYYYMMDD-<摘要>.md` か（拡張子は `.md` 単独）
- [ ] `date`、ファイル名先頭、`仕訳/YYYY/MM/DD` タグの3箇所が同じ仕訳発生日を指しているか
- [ ] `ID` と `created` は記帳した現在時刻（JST）か
- [ ] `debit` と `credit` の合計が一致しているか
- [ ] `debit`/`credit` がYAMLリストとして正しい形式か（`- category:` と `  amount:` のインデント）
- [ ] tagsに使われたすべての費目が `費目/<階層>` で列挙されているか
- [ ] 折半対象なら `仕訳/折半` タグを追加したか
- [ ] 費目が既存表記と揃っているか
