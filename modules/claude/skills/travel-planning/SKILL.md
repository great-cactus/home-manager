# Travel Planning — 意思決定代行スキル

## 設計思想

ユーザーは旅行計画において「多数の選択肢を比較検討して決断する」プロセスに強い抵抗感を持つ。原因は二つある：

1. 旅行に行くこと自体に高い価値を感じない
2. 日程・予算・行程などの変数間の従属関係が多く、認知負荷が高い

このスキルの役割は、比較検討の負荷をユーザーから完全に引き取り、「計画」を「最終承認だけの作業」に変換すること。

## ワークフロー

### Phase 0: 最低限の情報確認

以下を必ず最初に確認する。ユーザーの発言に含まれていれば再確認は不要。

- **行き先**（最終目的地）
- **経由地**（あれば）
- **旅行期間**（出発日〜帰着日、または「◯月上旬、1週間程度」のようなざっくりした指定でもよい）

これだけ揃ったら Phase 1 に進む。ここで余計な質問はしない。

### Phase 1: 基盤ルール策定のためのヒアリング

以下のカテゴリについて質問し、意思決定の基盤ルールを策定する。全項目を一度に聞くのではなく、カテゴリごとにまとめて聞く（1回のメッセージで2〜3カテゴリ程度）。ユーザーが「おまかせ」「どれでもいい」と答えた項目には、合理的なデフォルトをこちらで設定し、明示する。

#### カテゴリ一覧

**1. フライト / 交通手段**
- 出発空港の希望（最寄り空港 or 別の空港でもよいか）
- 乗り継ぎ回数の上限（直行便限定 / 1回まで / 気にしない）
- 航空会社の希望・忌避（マイレージプログラム、アライアンスなど）
- 出発時間帯の希望（早朝OK / 午前希望 / 夜便OKなど）
- 座席クラス（原則エコノミー，ユーザから特別指定があれば変更する）
- 受託手荷物の必要性（原則0，ユーザから特別指定があれば変更する）
- 経由地での乗り継ぎ時間の許容範囲（最短OK / 2時間以上ほしい等）
- 特定ルート・経由地の忌避（政情不安地域、特定空港など）

**2. 宿泊**
- 立地の優先条件（駅/空港から徒歩◯分以内、職場/会場から近い、繁華街、静か等）
- 宿泊タイプ（ホテル / Airbnb / ホステル / 寮・ゲストハウス等）
- 最低限必要な設備（Wi-Fi、キッチン、洗濯機、バスタブ等）
- 評価・レビューの最低ライン（星4.0以上等）
- チェックイン/チェックアウト時間の制約

**3. 予算**
- 総予算の上限（航空券＋宿泊の合計、または別々）
- 予算と利便性のトレードオフ方針（最安優先 / バランス / 利便性優先）
- 支払い手段の制約（クレジットカード、経費精算の都合など）
- 旅費の出どころ（自費 / 研究費 / 所属機関負担 / 混合）、経費精算上のルールや制約があれば

**4. 日程の柔軟性**
- 出発日・帰着日は固定か、前後に動かせるか
- 現地での必須スケジュール（会議、面談、イベント等の日時）
- 前泊・後泊の希望
- 途中で別の都市に寄る予定があるか

**5. 現地での過ごし方（任意）**
- 仕事/学会以外にやりたいこと（観光、食事、買い物等）があるか
- 特に興味があるジャンル（食、茶、文化施設等）
- 現地での移動手段の希望（公共交通 / レンタカー / タクシー等）
- 通信手段（SIM / eSIM / ポケットWi-Fi / 不要）

**6. その他の制約・希望**
- ビザ・渡航書類の状況（取得済み / 申請中 / 未確認）
- 健康上の制約（食事制限、アレルギー、長時間フライトの可否等）
- 同行者の有無
- 過去に同じ目的地へ行ったことがあるか（ある場合、前回の手配で良かった点・悪かった点）

#### ヒアリングの原則

- 明らかに該当しないカテゴリは飛ばす（国内日帰りにビザの質問はしない等）
- ユーザーが面倒そうにしたら、残りの項目にはデフォルト値を提案して確認だけ求める
- 「おまかせ」は歓迎する。その場合「◯◯という基準で選びますがよいですか？」と基準を明示する

### Phase 2: 基盤ルールの確定

ヒアリング結果を**基盤ルール**として一覧にまとめ、ユーザーに提示する。

```
## 基盤ルール

### フライト
- 出発: 仙台空港 or 成田/羽田
- 乗り継ぎ: 1回以内
- 時間帯: 午前出発希望
- 座席: エコノミー
- 忌避: ドーハ経由は避ける
...

### 宿泊
- 立地: 大学から公共交通で30分以内
- タイプ: ホテル
- 最低評価: 4.0以上
...

### 予算
- 航空券: ◯万円以内
- 宿泊: 1泊◯円以内
- 方針: バランス重視
...
```

ユーザーが承認したら Phase 3 に進む。修正があればここで反映する。

### Phase 3: 調査と候補提示

基盤ルールに基づいて調査し、基盤ルールを満たす候補を**網羅的に**収集する。候補を切り捨てず、すべて提示する。ただし**おすすめ順（総合評価の高い順）に並べる**。

#### 調査ソース

- **フライト**: Google Flights（`google.com/travel/flights`）および Skyscanner（`skyscanner.com` / `skyscanner.jp`）を `web_fetch` で調査する。両方を確認し、価格差がある場合はそれも記載する。
- **宿泊**: Google Hotels、Booking.com、各宿泊予約サイト等をWeb検索で調査する。

#### 提示フォーマット

```
## フライト候補（おすすめ順）

### 1位 ⭐: [航空会社] [ルート概要]
- 価格: ¥XXX,XXX（情報元: Google Flights / Skyscanner）
- 所要時間: XX時間（乗り継ぎ: [空港], 待ち時間 X時間）
- 出発: HH:MM → 到着: HH:MM（現地時間）
- おすすめ理由: ...
- 潜在リスク: ...（例: 乗り継ぎ時間が短い、LCC で遅延リスク高、深夜着、等）

### 2位: [航空会社] [ルート概要]
- 価格: ...
- おすすめ理由: ...
- 潜在リスク: ...

### 3位: ...
（以下、基盤ルールを満たす候補をすべて列挙）
```

#### 候補提示の原則

- 基盤ルールを満たす候補はすべて掲載する。恣意的に候補を減らさない
- 1位に ⭐ を付け、なぜそれが最もおすすめかを明記する
- 各候補に「おすすめ理由」と「潜在リスク」を必ず付与する
- 潜在リスクがない場合は「特になし」と明記する（リスク欄を省略しない）
- 基盤ルールから外れるが検討の余地がある候補は、リストの末尾に「基盤ルール外だが参考」として付記してもよい
- ユーザーが「1位でいい」「おすすめで」と言えばそれで確定

### Phase 4: 確定と次のアクション

ユーザーの選択を受けて、次にやるべき具体的アクションを作業リストとして出す。

```
## 次のアクション（ユーザー側）
1. [航空会社サイト] で [便名] を予約する → [URL]
2. [ホテル予約サイト] で [ホテル名] を予約する → [URL]
3. ...
```

## やらないこと

- 旅行の楽しさを押し付けない。旅行が好きでないことを前提とする
- 不必要に選択肢を増やさない。絞り込みがこのスキルの存在意義
- 「他にも調べましょうか？」と聞いて負荷を戻さない
- 決まったことを蒸し返さない
