# AkiraのObsidian × AI協同ルール

## 根本思想

AkiraのObsidian vaultは「自分の思考をシャープにする場」である。したがって：

1. **AIノートの乱立は禁忌**。AIが書いたmdファイルがvault本体に増殖することは、Akiraの思考プロセスを濁らせる。
2. **AI生成テキストとAkiraのテキストは原則として視覚的に明確に区別される必要がある**。ファイルを開いた瞬間、「どこからどこまでがAIか」が一目で分かること。**ただしこの分離は絶対則ではなく、後述の「許可付き直接編集」では適用されない**（§許可付き直接編集を参照）。
3. **AIはvaultに書き込む権限を持つが、書ける場所と形式は厳格に制限される**。下記の3経路以外には何があっても書かない。
4. **AI生成ノートは検証可能でなければならない**。AIが外部情報源（論文・書籍、webサイト、vault内の他のノート）を参照して書いた場合、その出典を必ず明記する。逆に、論理的推論や明らかな一般知識のみで完結する記述には出典は不要であり、無理にでっち上げてはならない。

### 分離原則の例外について（重要な前提）

思想2の「AIテキストとAkiraテキストの分離」は、**AIが頼まれてもいないのに勝手に書いたテキストが本人の思考に紛れ込む事故を防ぐ**ためのものである。したがって、Akiraが**校閲・整形・修正を明示的に依頼し、その結果を確認・承認したうえで受け入れる**場合には、この懸念は当てはまらない。これは論文校閲や原稿作成における通常のワークフロー（校閲者が直接本文を直し、著者が承認した時点でそれは著者の文章になる）と同型であり、Akiraの責任のもとAI由来の変更を本人のテキストとして受け入れることが意図されている。この場合、AIは痕跡を残さず本文を直接編集してよい（§許可付き直接編集）。

## 書き込み場所の三経路

AIがvaultに書き込み・編集してよいのは次の3経路のみ。それ以外（`Output/`配下への新規mdファイル無断作成、`Input/`, `GTD/`, `Bookkeeping/`、vaultルート直下の自作ノート本文の無断書き換えなど）にAIが勝手にファイルを作成したり既存テキストを書き換えたりしてはならない。

| 経路 | 用途 | 条件・区別の仕組み |
|---|---|---|
| (1) `.output_by_ai-agent/<topic>.md` | AIが完結した文章を生成する場合（チャットのまとめ、調査結果、構造化メモ） | Obsidianは隠しディレクトリ（`.` で始まる）を検知しないので、vault本体の検索/グラフビュー/Dashboardから自動的に隔離される |
| (2) 既存ノート内の `> [!llm]` callout | ちょっとした補足（数行〜短い段落） | Akira定義のCSSスタイルでLLM由来と一目で判別できる |
| (3) `Output/`配下ノートの**本文を直接編集**（許可付き直接編集） | ユーザが明示的に依頼した校閲・整形・修正 | **その都度の明示許可が必須**。許可された範囲の編集は痕跡を残さず本文に溶け込ませる（§許可付き直接編集） |

経路(1)(2)は「AIが自発的または依頼で*付け加える*」ための経路で、分離原則（思想2）が効く。経路(3)は「ユーザ依頼でAIが本文を*直す*」ための経路で、分離原則は適用されない。

### 判断フロー

書き込み・編集リクエストを受けたら、次の順で判定する：

1. **ユーザが既存ノートの本文の校閲・整形・修正を、その会話の中で明示的に依頼したか？**（「校閲して直して」「フォーマット整えて」「本文を修正して」「このノートを直して」等）
   - Yes かつ対象が `Output/` 配下 → 経路(3) 許可付き直接編集（§許可付き直接編集の条件を満たすこと）
   - Yes だが対象が `Input/`, `GTD/`, `Bookkeeping/` → 直接編集は不可。GTD/Bookkeepingは専用スキルへ、Inputは編集対象外（補足が要るなら経路(2)）
2. **ユーザが「まとめて／保存して／ノートにして／サマリにして」と明示的に指示したか？**
   - Yes → 経路(1) `.output_by_ai-agent/<topic>.md` を新規作成
3. **既存ノートを編集中で、AIから補足を加えたいか？（直接編集の依頼ではない）**
   - 数行〜1段落程度の短い補足 → 経路(2) そのノートに `> [!llm]` ブロックで挿入
   - それ以上の分量（長い解説、表、コードブロックなど） → 経路(1) `.output_by_ai-agent/<topic>.md` に切り出し、元ノートから `[[<topic>]]` でリンク
4. **ユーザの明示的依頼でなく、補足とも違うか？**
   - vaultに書き込まず、チャットで返すだけ

判断に迷ったら**書き込まない／直接編集しない**のがデフォルト。Akiraは「AIに頼んでいないことをAIがやる」のを嫌う（user memory に明記あり）。特に経路(3)は、明示依頼が無い限り絶対に発動しない。校閲結果を本文に反映してよいか曖昧なときは、経路(2)の `> [!llm]` で指摘だけ示すか、チャットで返す。

## `.output_by_ai-agent/<topic>.md` の書き方

### ファイル名

`<topic>.md`。既存のPERMANENTノートと同じ命名規則。日付プレフィックスもIDプレフィックスもつけない。`title` フィールドと一致させる。

### Frontmatter

PERMANENTノート互換のfrontmatter。ただし種別タグは `PERMANENT` ではなく **`WRITTEN_BY_AI`** を使う。これによりDashboard.mdのPERMANENTヒートマップ等への混入を防ぐ。

```yaml
---
ID: 202605161430
created: 2026-05-16 14:30
title: <topic>
aliases: []
tags: [ 2026/05/16, WRITTEN_BY_AI, <内容タグ...> ]
---
```

- `ID`：`YYYYMMDDhhmm`（12桁、JSTローカル時刻）
- `created`：`YYYY-MM-DD HH:mm`（JST）
- `title`：ファイル名 `<topic>` と一致させる
- `aliases`：空配列 `[]`（実例の慣例に揃える）
- `tags`：**インライン記法** `[ a, b, c ]`（Output系の慣例に揃える）

### タグ規則（重要）

タグ順序は **`obsidian-personal-zettelkasten` スキルのタグ規則を踏襲**する：

```
[ <日付YYYY/MM/DD>, WRITTEN_BY_AI, <内容タグ...> ]
```

- **日付タグ `YYYY/MM/DD` は必須**。スラッシュ区切り、ゼロ埋め。忘れずに付ける
- **`WRITTEN_BY_AI` も必須**。これが無いとAI生成物として扱われない
- 内容タグはAIが**自由に**付与してよい。階層タグも可（例：`laminar_flame_theory`, `Project/SPReAD_grant`, `Fuel/NH3`）
- 既存vaultで使われているタグと整合させたいなら、書き込む前に `obsidian_simple_search` で関連タグの実例を確認して揃える

### 本文

- 通常のmarkdownでOK。本文内では `> [!llm]` で囲む必要は**ない**（ファイル全体がAI生成と既にfrontmatter `WRITTEN_BY_AI` で示されているため）
- **AIノート同士の相互リンクは自由に張ってよい**：`[[<other-topic>]]` でwikilink解決に任せる。複数のAIノートで網を作るほうがZettelkasten的に望ましい
- vault本体（`Output/`配下のノートなど）への参照リンクも `[[<note>]]` で普通に書ける。ただし**本体側からこちらへの逆リンク（バックリンク）はObsidian上には現れない**（隠しディレクトリは検出されないため）。これは意図された設計：AIノートはvault本体を「汚染」しない
- 長くなりそうなら1ファイルに詰め込まず、テーマで分割して相互リンクするほうがよい
- **外部情報源を参照した本文記述には、その場で `[1]`, `[2]` のような連番のインライン参照番号を入れる**。番号は本文末尾の `## References` セクションのエントリに対応する。Vault内ノートへのリンクは `[[<note>]]` をそのままインラインで使い、連番には含めない

### References セクション（外部情報源を参照したときのみ必須）

本文で外部情報源を参照した場合、本文末尾に `## References` セクションを設けて列挙する。**参照したものが無いノート（純粋な論理展開、ユーザとの議論のみで完結する内容）ではこのセクションを省く**。出典が無いのに無理にそれらしいものを書くのは厳禁。

#### 論文・会議録・書籍 — Elsevier num-with-title スタイル

雛形：

```
[1] A.B. Author, C.D. Author, E.F. Author, Title of the paper, J. Name Abbr. Vol (Year) pp1–pp2. https://doi.org/10.xxxx/yyyy.
[2] A.B. Author, C.D. Author, Title of conference paper, in: Proc. Conf. Name, City, Year, pp. xx–yy. https://doi.org/...
[3] A.B. Author, Book Title, Xth ed., Publisher, City, Year, pp. xx–yy.
```

具体例：

```
[1] R.E. Mitchell, A.F. Sarofim, L.A. Clomburg, Experimental and numerical investigation of confined laminar diffusion flames, Combust. Flame 37 (1980) 227–244. https://doi.org/10.1016/0010-2180(80)90094-3.
```

ルール：

- 著者は `A.B. Author`（イニシャル＋姓、イニシャル間にスペースなし、`A.B.` で1まとまり）。3名までは全列挙、それ以上は `A.B. First et al.`
- 著者間の区切りはコンマ。最終著者の前に "and" は入れない（Elsevier慣例）
- **タイトルに引用符を付けない**。文頭のみ大文字（タイトルケースではなく sentence case）
- ジャーナル名は標準略記（不明なら正式名で可）。斜体強調は付けない
- 巻号・年・ページは `Vol (Year) pp1–pp2.` の順で詰める。号数は通常省く（Elsevierでは巻のみで一意なら号は省略）
- ページ範囲のハイフンは en-dash `–`
- **DOI は存在すれば必ず付ける**。形式は `https://doi.org/10.xxxx/yyyy.` のフルURL。`doi:` 接頭辞は付けない。DOI が無い場合のみ `[Online]. Available: <URL>` で代用
- 書籍は `Publisher, City, Year` の順

Zoteroで管理している論文を参照した場合は、Zotero MCP（`zotero_item_metadata`）でメタデータを取得してこの形式に整形する。citekey や Zotero URL の併記は不要。

#### Webサイト・オンライン記事

```
[4] Page Title, Site Name, https://... (accessed YYYY-MM-DD JST).
```

- アクセス日（その情報を取得した日のJST）を必ず付ける
- ブログ記事や preprint サーバ（arXiv等）でジャーナル論文に相当するものは、上の論文形式に揃えても可（arXivなら `arXiv:xxxx.xxxxx` を DOI 相当として末尾に置く）

#### Vault内の他のノート

References セクションに含める必要は**ない**。本文中で `[[<note>]]` のwikilinkを張れば十分（Obsidianのバックリンク機能で追跡できる）。ただし「議論の前提として依拠した自分のノート」を明示的に列挙したい場合は、References の末尾に：

```
[Internal] [[<note-a>]], [[<note-b>]]
```

の形で並べてよい。

#### 参照番号と本文の対応

本文中で `[1]` のように引いた番号は、必ず References セクションの同番号と一致させる。番号の振り直しを避けるため、本文を書いてから References を確定するか、References を先に固めて本文を書くかのどちらかで運用する。

## `> [!llm]` callout の書き方

既存ノート（PERMANENT、LITERATURE、DailyNoteなど）にAIが補足を加える場合：

```markdown
> [!llm]
> 補足の内容を書く
> 複数行も可
```

複数段落にわたる補足：

```markdown
> [!llm]
> 1段落目の内容
>
> 2段落目（`>` だけの空行で段落を分ける）
```

タイトル付きcallout：

```markdown
> [!llm] 補足: 関連する先行研究
> 本文...
```

折りたたみ可能なcallout（長めの補足を最初は閉じておきたい場合）：

```markdown
> [!llm]- 詳細な背景説明
> 折りたたまれた状態で表示される本文
```

CSSによりLLM由来と即座に判別できる。**この記法を省略してAI生成テキストを既存ノート本文にそのまま書き込むのは絶対に禁止**。これがAkiraの「AIテキストとの境界」を維持する唯一の手段である。

「ちょっとした補足」の目安：おおむね数行〜1段落程度。それを超えるなら `.output_by_ai-agent/` に切り出してリンクで参照する。

### callout 内の出典記載

callout 内で外部情報源を参照した場合、末尾に `_Source: ..._` 行を1つ入れる。形式は References セクションと同じ Elsevier num-with-title スタイル（callout 内は1件で済むことが多いので連番 `[1]` は省いてよい）：

```markdown
> [!llm] 関連する先行研究
> Walker らは類似条件で C–C 結合の選択的解離を報告している。
>
> _Source: J. Walker, A.B. Smith, Selective C–C bond cleavage in polystyrene under nanosecond pulsed plasma, ACS Sustain. Chem. Eng. 12 (2024) 1234–1242. https://doi.org/10.1021/acssuschemeng.4xxxxxx._
```

Vault内ノートを参照した場合：

```markdown
> [!llm]
> 補足の内容
>
> _Source: [[<note>]]_
```

複数の出典を引いた callout なら：

```markdown
> [!llm]
> 補足の内容
>
> _Sources:_
> _[1] A.B. Author, C.D. Author, Title, J. Name 12 (2024) 1234–1242. https://doi.org/..._
> _[2] [[<note>]]_
```

**参照したものが無い純粋な論理的補足（自明な推論、用語の言い換え、構造の整理など）には `_Source:_` 行は不要**。出典が無いのに「AI inference」などと書く必要もない（callout であること自体がAI由来であることを示しているため）。

## 許可付き直接編集（経路3）

ユーザが**その会話の中で明示的に**校閲・整形・修正を依頼した場合に限り、AIは `Output/` 配下ノートの**本文を直接編集**してよい。この編集は痕跡を残さず本文に溶け込ませる（`> [!llm]` で囲まない、フラグも立てない、chat報告も不要）。校閲者が本文を直し著者が承認した時点でそれが著者の文章になる、という通常の校閲・原稿作成ワークフローと同型であり、Akiraの責任のもとAI由来の変更を受け入れることが意図されている。

### 発動条件（すべて満たすこと）

1. **その都度の明示許可**：ユーザがその会話で本文の校閲・整形・修正を明示的に依頼している。許可は会話をまたいで持ち越さない。前の会話やデフォルト設定での許可は無効で、毎回あらためて明示依頼が要る。
2. **対象は `Output/` 配下のみ**：`Input/`, `GTD/`, `Bookkeeping/`、vaultルート直下のノートは対象外。これらは従来通り保護され、直接編集してはならない（GTD/Bookkeepingは専用スキル、Inputは外部ソースのメタデータ）。
3. **編集の性質が校閲・整形・修正であること**：誤字脱字、表記ゆれの統一、フォーマット整形、数式・用語・事実の誤りの修正、文章の自然化など、「既にある本文を直す」作業に限る。ユーザが依頼していない新規の主張・解説・節の追加は経路(3)ではなく経路(1)(2)で行う（依頼された修正のついでに本文を膨らませない）。

いずれか1つでも満たさないなら経路(3)は使わず、`> [!llm]`（経路2）かチャット返答にとどめる。曖昧なら直接編集しない。

### 直接編集の手順

1. **対象ノートを読む**：`obsidian_get_file_contents` で現在の本文を取得し、直すべき箇所を特定する。
2. **編集を実行**：`obsidian_patch_content`（heading/block/frontmatter 指定での置換）または該当箇所の書き換えで、本文を直接修正する。`> [!llm]` で囲まない。
3. **frontmatterは原則保持**：ユーザが整形を依頼した場合を除き、既存frontmatter（ID, created, title, tags 等）は変更しない。`WRITTEN_BY_AI` タグやAI由来フラグは**付けない**（このノートはあくまでAkiraのノートであり、AIノートではない）。
4. **外部情報源の扱い**：校閲の根拠として外部情報源を参照しても、本文がAkiraの文章である以上、AIの出典注記（`[1]` 連番や `_Source:_`）を本文に挿入する必要はない。ただし「この修正は外部資料に基づく」という事実をユーザに伝えるべき場合は、編集後に**チャットで**根拠を述べる（本文には残さない）。本文中に元から引用・参考文献の体裁がある場合は、その体裁に合わせて直す。
5. **変更内容をチャットで要約**：何をどう直したかを簡潔にチャットで報告する。痕跡をノートに残さないぶん、ユーザが承認・追跡できるよう編集サマリを会話側に置く。

### 直接編集で守ること

- **依頼された範囲を超えない**：「式を直して」と言われたら式を直す。頼まれていない別の段落の主張を書き換えたり、新しい考察を足したりしない。
- **本文の声を保つ**：Akira自身の文体・記号慣習・既存の用語選択を尊重する。校閲はAkiraの文章をAkiraの声のまま正すことであり、AIの文体に置き換えることではない。
- **数式はmath-notationスキルに従う**：式の整形・修正を伴う場合、立体/斜体・分数・変数定義の規約は `math-notation` スキルを優先する。
- **不確実な修正は勝手に確定しない**：事実・数値・式の正誤に確信が持てない箇所は、本文を書き換えるのではなく `> [!llm]` で疑問として示すか、チャットで確認する。誤った「修正」を本文に溶け込ませるのが最も危険。

## 書き込み手順

1. **書き込み先を判定**（上の判断フロー）。校閲・整形・修正の明示依頼かつ `Output/` 配下なら経路(3) へ。
2. **書き込み権限を確認**：ユーザが明示的に書き込み・編集を求めているか、または既存ノート編集の文脈で補足が自然か。曖昧なら1回だけ確認する。経路(3)（直接編集）は明示依頼が無ければ発動しない。
3. **外部情報源の有無を確認**：書こうとしている内容に外部参照（論文・Web・他ノート）があるか。あれば出典情報を控えておく（経路1・2では本文/calloutに明記、経路3では本文には残さずチャットで補足）
4. **`.output_by_ai-agent/<topic>.md` を作る場合**（経路1）：
   - 現在時刻（JST）からID（12桁）/ created / 日付タグ `YYYY/MM/DD` を生成
   - frontmatterを上記スキーマで組み立てる（インライン記法のtags、順序「日付→WRITTEN_BY_AI→内容」）
   - 内容タグはトピックに応じて自由に付与。既存タグと整合させたいなら事前に検索
   - 本文を書く際、外部情報源の言及には `[1]` `[2]` のような連番を入れる
   - 外部情報源を引いたなら本文末尾に `## References` セクションを Elsevier num-with-title 形式で組む
   - `mcp-obsidian:obsidian_append_content` で `.output_by_ai-agent/<topic>.md` を新規作成
5. **既存ノートに `> [!llm]` で追記する場合**（経路2）：
   - 既存frontmatterは触らない
   - 適切な位置に `> [!llm]` ブロックを挿入
   - 外部情報源を引いたなら callout 末尾に `_Source: ..._` 行を入れる
   - `mcp-obsidian:obsidian_patch_content`（位置指定）か `obsidian_append_content`（末尾追記）を使う
6. **`Output/` 配下ノートを直接編集する場合**（経路3）：
   - §許可付き直接編集の発動条件3つをすべて満たすことを確認
   - `obsidian_get_file_contents` で本文を取得し、修正箇所を特定
   - `obsidian_patch_content` 等で本文を直接書き換え（`> [!llm]` で囲まない、AI由来フラグを付けない）
   - frontmatterは原則保持
   - 何をどう直したかをチャットで要約報告

## 禁止事項

- ❌ AIが勝手に `Output/` 配下にPERMANENT/LITERATUREノートを作成する
- ❌ `WRITTEN_BY_AI` タグ無しで `.output_by_ai-agent/` にファイルを作る
- ❌ 日付タグ `YYYY/MM/DD` を忘れる
- ❌ 既存ノート本文に `> [!llm]` 無しでAIテキストを書き加える
- ❌ AI生成ノートに `PERMANENT` タグをつける（Dashboardのヒートマップが汚染される）
- ❌ `Input/` 配下のファイルを編集する（外部ソースのメタデータ。経路3の直接編集でも対象外）
- ❌ `GTD/` `Bookkeeping/` 配下に書き込む・直接編集する（それぞれ専用スキルがある。経路3の対象外）
- ❌ ユーザが求めていないのに勝手に保存する
- ❌ **明示依頼が無いのに `Output/` 配下ノートの本文を直接書き換える**（経路3は毎回の明示許可が必須。許可が無ければ `> [!llm]` かチャット返答）
- ❌ **前の会話や「デフォルトで許可」を根拠に直接編集する**（許可は会話をまたいで持ち越さない）
- ❌ **校閲・整形・修正の依頼に乗じて、頼まれていない新規の主張・解説・節を本文に書き加える**（追加は経路1・2へ）
- ❌ **正誤に確信が持てない修正を本文に溶け込ませる**（不確実なら `> [!llm]` かチャットで確認）
- ❌ **外部情報源を参照したのに `## References` セクション（または callout の `_Source:_` 行）を省く**（経路1・2の場合。経路3の本文には出典注記を入れず、必要ならチャットで補足）
- ❌ **存在しない論文・URLをそれらしく捏造する**。Zotero MCP / web_search / 既存ノート検索で実在確認できない出典は書かない。曖昧なら出典の代わりに「該当する一次資料が手元に無い」とユーザにチャットで返す
- ❌ **Elsevier num-with-title 形式以外の独自フォーマットで論文を引用する**（IEEE風の引用符付きタイトル、citekey 単独、URL 単独、APA 風など）

## 既存スキルとの関係

- **`obsidian-personal-zettelkasten`**：Akiraが自分で書くPERMANENT/LITERATUREノートのルール。AIがこれらに*自発的に書き足す*のは必ず `.output_by_ai-agent/` 側か `> [!llm]`。ただし**ユーザが校閲・整形・修正を明示依頼した場合は経路3でこれらの本文を直接編集してよい**（その際もZettelkastenの形式規約は尊重する）
- **`math-notation`**：経路3で数式の整形・修正を行う場合、立体/斜体・分数・変数定義の判断は本スキルより `math-notation` を優先する
- **`scientific-english-proofreading` / `email-english-proofreading`**：英文校閲の実体的判断はこれらのスキルが担う。本スキルは「校閲結果をvaultのどこにどう書き込む／反映するか」の経路だけを規定する。両者を併用してよい
- **`obsidian-personal-gtd`**：タスク管理。AIが代行する場合でも書き込み先は `GTD/` であり、本スキルの対象外（経路3でも編集しない）
- **`obsidian-personal-bookkeeping`**：家計簿。同様に本スキルの対象外

「Zettelkasten/GTD/Bookkeepingのどれにも当てはまらないが、AIが何かをvaultに書き込みたい」場合、および「ユーザ依頼でOutput/配下ノートを直接校閲・整形・修正したい」場合に本スキルが効く。

## 書き込み前の最終チェックリスト

mcp-obsidianの書き込み・編集ツールを呼ぶ直前に確認：

**全経路共通**
- [ ] 書き込み・編集先は経路(1) `.output_by_ai-agent/<topic>.md`、経路(2) `> [!llm]` callout、経路(3) `Output/`配下本文の直接編集、のいずれかか
- [ ] ユーザの明示的依頼か、自然な補足の文脈か
- [ ] `Input/`, `GTD/`, `Bookkeeping/` を編集していないか

**経路(1) `.output_by_ai-agent/` の場合**
- [ ] frontmatterに `WRITTEN_BY_AI` と日付タグ `YYYY/MM/DD` が両方入っているか
- [ ] tagsは**インライン記法** `[ ... ]` で、順序は「日付 → WRITTEN_BY_AI → 内容」になっているか
- [ ] ID/createdはJST現在時刻から正しく生成したか
- [ ] PERMANENTタグを誤ってつけていないか

**経路(2) `> [!llm]` の場合**
- [ ] 補足は `> [!llm]` で囲まれているか
- [ ] 既存frontmatterを触っていないか

**経路(3) 許可付き直接編集の場合**
- [ ] その会話でユーザが校閲・整形・修正を**明示的に依頼**したか（持ち越し許可は無効）
- [ ] 対象は `Output/` 配下か
- [ ] 編集の性質は校閲・整形・修正に収まっているか（頼まれていない新規主張を足していないか）
- [ ] 本文を `> [!llm]` で囲んでいない／AI由来フラグやタグを付けていないか
- [ ] frontmatterを原則保持しているか
- [ ] 正誤に確信が持てない修正を本文に溶け込ませていないか（不確実なら `> [!llm]` かチャット確認）
- [ ] 数式を直すなら `math-notation` 規約に従っているか
- [ ] 何をどう直したかをチャットで要約報告する準備があるか

**外部情報源を引いた場合（経路1・2）**
- [ ] `## References` セクション（または callout 内の `_Source:_` 行）に Elsevier num-with-title 形式で出典が書かれているか
- [ ] 本文中の参照番号 `[1]`, `[2]` ... と References セクションの番号が一致しているか
- [ ] 論文のタイトルに引用符が付いていないか（Elsevier流：引用符なし、sentence case）
- [ ] 著者表記が `A.B. Author` 形式（イニシャル＋姓、イニシャル間スペースなし）になっているか
- [ ] 巻号年ページが `Vol (Year) pp1–pp2.` の順になっているか
- [ ] 論文の出典に DOI（`https://doi.org/...` のフルURL形式）が付いているか
- [ ] Web 出典にアクセス日 `(accessed YYYY-MM-DD JST)` が付いているか
- [ ] 出典をでっち上げていないか（Zotero / web_search / vault検索で実在確認したものだけ書く）

このチェックを通ったうえで書き込む・編集する。
