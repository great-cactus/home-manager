# Storyline Outliner

散文やチャットを、ストーリーラインが明確に見えるアウトラインに変換するためのスキル。

## このスキルの役割

書き手が散文を書いているとき、論理構造は文章の中に埋め込まれているが、その骨格が読み手・書き手自身にとって見えにくい。アウトライナー化はその骨格を取り出して可視化する作業である。

要約（abstractive summary）が「内容を短くする」のに対し、アウトライナー化は「構造を抽出する」。長さよりも、ストーリーラインの可視性が優先される。

---

## アウトライナーが満たすべき原則

以下の原則を満たすアウトラインを生成する。原則同士は相補的なので、すべてを同時に満たすよう調整する。

### 1. アウトライナーの定義

アウトライナーは、ストーリーラインを確認するための箇条書きの集まりである。インデントによって階層構造を表現する。

### 2. 1箇条1トピック

各箇条はちょうど1つのトピックを含む。

- 同じトピックを複数の箇条に書かない（重複禁止）
- 1つの箇条に複数のトピックを詰めない

### 3. 各箇条は命題として書く

各箇条は体言止めやキーワード列ではなく、主張・命題として書く。「〜である」「〜が必要だ」「〜が問題となる」のように、それ自体で主張として読める文にする。

体言止めだとストーリーラインが見えない。箇条がトピック名のラベルになってしまい、書き手が何を言いたいのかが伝わらないからである。

### 4. 同階層の並列性

同じ階層の箇条は、上位箇条に対して**並列関係**にある。具体的には次の3つを満たす。

- **MECE性**：上位箇条を漏れなく・重複なく分解する
- **役割の一貫性**：同階層の箇条はすべて、上位箇条に対して同じ種類の関係（根拠／構成要素／時系列ステップ／例示／対比など）を持つ
- **粒度の一貫性**：同階層では情報の粒度（詳細度）を揃える

「役割の一貫性」が崩れる典型例は、ある箇条の下に「根拠」「具体例」「反論」が混ざること。これらは別の役割なので、混在させると読み手は何を読まされているのか分からなくなる。

### 5. 上位→下位は抽象→具体に推移

上位の箇条から下位の箇条にかけて、抽象から具体、全体から詳細に推移する。下位は必ず上位を支える（補足・分解・例示・根拠提示）形になる。

### 6. 順序にも意味を持たせる

同階層の箇条の並び順にも意味を持たせる：時系列、因果関係、重要度、論理的依存など。並列だからといって順不同にしない。

### 7. 上位だけで筋が通る（要約可能性）

最上位の箇条だけを順に読んだとき、それだけで筋の通った要約として読める。第2階層まで読めば概要として読める、第3階層まで読めば詳細が分かる、というように層が重なる。

逆に言えば、ある階層を切り落としても上の階層では物語が成立する。

### 8. すべての箇条がトップ主張に貢献する

最上位のテーマ・主張に対して、すべての箇条が何らかの形で貢献している。寄与しない箇条は削るか、別の場所に移す。

### 9. 階層の深さの目安

3〜4階層を目安にする。深すぎると見通しが悪くなる。どうしても深くなる場合、上位の構造を見直す（並列の取り方が間違っている可能性が高い）。

## 散文・チャットからの変換手順

アウトライナーは「文章全体を1文に圧縮したもの」ではなく「ストーリーラインを複数の命題の連なりで表現したもの」である。この区別が手順の根幹にある。

入力テキストを読み、以下の手順で構造を抽出する。手順は線形ではなく、Step 5 の自己チェックで戻ることが多い。

### Step 1: 文章のテーマ・性格を把握し、必要ならタイトルとして記録する

入力テキストが「何の文章か」（申請書、論文ドラフト、ブログ記事、研究メモ、議事録など）と「全体として何を扱うか」を把握する。

把握したテーマは Markdown 見出し（`# タイトル`）として記録する。タイトルが不要な短いメモなどでは省略してもよい。

**ここで「全体を1文に圧縮しよう」としない**こと。ストーリーラインは1文ではなく、複数の命題の連なりとして次の Step で展開される。

入力タイプ別のヒント：

- **論文・記事の散文**：論文タイトル・記事タイトルがそのままタイトルになる。
- **LLMとのチャット**：チャットのタイトルがあればそれを使う。なければ、何の検討をしているかを1〜2語で記述する。
- **メモや断片**：書き手が何のために書いたかを推定する。

### Step 2: 第1階層の軸（役割）を選ぶ

第1階層をどんな構成で展開するかの「軸」を決める。これが第1階層の役割の一貫性を担保する。よくある軸：

- **問題と解決**：問題提起 → 既存手法の限界 → 提案 → 検証 → 含意
- **時系列**：過去 → 現在 → 未来、または準備 → 実行 → 振り返り
- **構成要素分解**：要素A／要素B／要素C
- **因果連鎖**：原因 → メカニズム → 帰結 → 含意
- **論証構造**：主張 → 根拠 → 反証への応答 → 結論
- **対比**：観点1での比較／観点2での比較／観点3での比較

軸を決めずに書き始めると、役割が混在して並列性が崩れる。

### Step 3: 第1階層を3〜7個の命題で書き出す

選んだ軸に従って、第1階層の箇条を書き出す。**第1階層がストーリーラインの骨格そのもの**になる。

要件：

- 第1階層だけ読んで筋が通る（ストーリーラインとして読める）レベルにする
- MECE性：漏れなく重複なく
- 役割の一貫性：同じ役割で並ぶ
- 各箇条は命題として書く（体言止め禁止）

第1階層は通常3〜7項目になる。1個だけになるのは過圧縮のサイン（アンチパターン7参照）。

### Step 4: 各第1階層を第2階層以下に展開する

各第1階層箇条について、それを支える詳細を第2階層・第3階層に展開する。各階層内で「役割の一貫性」「粒度の一貫性」「MECE性」を保つ。

第2階層・第3階層の役割は、その親に対する役割であり、第1階層の役割とは独立に決められる。

### Step 5: 自己チェック（後述）を全階層にかける

チェックリストで全体を見直す。問題があれば該当箇所を修正し、必要なら上位の階層も見直す（局所修正で済まないことが多い）。

### Step 6: 順序を最適化

同階層の並びを、時系列・因果・論理的依存に従って整える。

---

## 自己チェックリスト

書いたあと、以下を順にチェックする。1つでも違反があれば修正する。

1. **命題チェック**：各箇条は体言止めではなく、命題として書かれているか
2. **1箇条1トピック**：1つの箇条に複数のトピックが入っていないか
3. **トピック重複なし**：同じトピックが別の箇条に書かれていないか
4. **MECE性**：同階層は上位を漏れなく重複なく分解しているか
5. **役割の一貫性**：同階層の箇条は同じ役割か（根拠と例の混在は禁止）
6. **粒度の一貫性**：同階層は詳細度が揃っているか
7. **抽象→具体の推移**：下位は上位より具体的になっているか
8. **第1階層の自立性**：第1階層だけを順に読んで、ストーリーラインとして筋が通るか
9. **第1階層の項目数**：第1階層は3〜7項目あるか（1個だけは過圧縮）
10. **トップ主張への貢献**：すべての箇条がタイトル/全体テーマに貢献するか
11. **順序の意味**：同階層の並びに意味があるか
12. **階層の深さ**：3〜4階層に収まっているか

## アンチパターンと修正例

### アンチパターン1：体言止め

❌

```
- 機械学習における過学習
  - 原因
  - 対策
```

✅

```
- 機械学習では過学習がモデルの汎化性能を損なう
  - 過学習はモデルが訓練データの偶然のパターンまで学習することで起こる
  - 正則化やデータ拡張で過学習を抑制できる
```

体言止めだと「過学習について何を言っているか」が分からない。命題化すると主張が見える。

### アンチパターン2：トピックの重複

❌

```
- ZFK-FKPP遷移はノック発生を予測する重要なメカニズムである
- ノックの予測にはZFK-FKPP遷移の理解が必要である
```

同じトピックが2回現れている。1つにまとめるか、片方を別の観点に書き換える。

### アンチパターン3：役割の混在

❌（第2階層に「根拠」と「例」と「反論」が混在）

```
- 提案手法は有効である
  - 既存研究で同様の効果が報告されている    ← 根拠
  - エンジン実験で誤差5%以内に収まった       ← 検証結果
  - 一部の研究者はこの手法に懐疑的である     ← 反論
```

✅（役割を統一）

```
- 提案手法の有効性は複数の検証結果で支持される
  - エンジン実験では誤差5%以内に収まった
  - シミュレーションでは既存手法より精度が向上した
  - 関連研究の結果とも整合する
```

「反論」を扱いたいなら、別の第2階層箇条を立てる：

```
- 提案手法には議論の余地が残る
  - 一部の研究者は適用範囲が限定的と指摘する
  - ...
```

### アンチパターン4：上位だけ読んで筋が通らない

❌

```
- ZFK-FKPP遷移について
  - 提案する
  - 検証する
```

最上位だけ読むと「ZFK-FKPP遷移について」しか分からない。

✅

```
- ZFK-FKPP遷移モデルによりノック発生時刻を予測できる
  - 既存のラジカル爆発モデルを拡張してZFK-FKPP遷移条件を導入する
  - エンジン実験データで予測精度を検証した結果、誤差5%以内であった
```

上位だけ読んで「何を主張し、それをどう検証したか」が伝わる。

### アンチパターン5：粒度の不揃い

❌

```
- 環境問題への対応が必要である
  - 化石燃料を再生可能エネルギーに切り替える
  - 太陽光パネルの発電効率は20%程度である   ← 急に細かい数値
  - リサイクルを促進する
```

太陽光パネルの数値だけ粒度が違う。第3階層に降ろすか、別の項目と粒度を揃える。

### アンチパターン6：階層が深すぎる

第5階層、第6階層と深くなる場合、上位の取り方が間違っているサイン。たとえば第2階層が抽象的すぎて、本来並列であるべきものが第3階層・第4階層に押し込まれている可能性が高い。一度第2階層を見直す。

### アンチパターン7：第1階層が1個だけ（過圧縮）

❌

```
- 提案手法は有効である
  - 既存研究の限界は〜
  - 提案手法は〜
  - 検証では〜の結果が得られた
```

第1階層が1個しかないと、ストーリーラインが見えなくなる（第1階層を読んでも分かるのは「提案手法は有効である」という抽象的な主張のみで、論証の流れが追えない）。これは「文章全体を1文に圧縮しよう」とした結果起こる典型的な失敗。

✅

```markdown
# 提案手法の有効性検証

- 既存手法には〜という限界がある
- 提案手法は〜の点で既存の限界を克服する
- 検証では〜の結果が得られた
- この結果は〜という含意を持つ
```

全体のテーマは Markdown 見出し（`#`）として書き、第1階層はストーリーラインを直接構成する複数の命題（通常3〜7個）として並べる。第1階層だけを読んで、文章のストーリーが追えることを目標にする。

## 出力フォーマット

Markdown箇条書きで出力する。インデントは**半角スペース2つ（shiftwidth=2）**を使う。タブや4スペースは使わない。

全体構造：

```markdown
# 文章のタイトル/全体テーマ（任意、省略可）

- 第1階層の命題1（ストーリーラインを構成する）
  - 第2階層の詳細
  - 第2階層の詳細
    - 第3階層の詳細
- 第1階層の命題2
  - 第2階層の詳細
- 第1階層の命題3
  - 第2階層の詳細
```

- タイトル（`#` 見出し）は文章の性格・全体テーマを示す。短いメモなどでは省略可
- 第1階層がストーリーラインの骨格。第1階層だけ読んでも文章の筋が通るのが目標
- 第2階層以下は各第1階層を支える詳細

箇条が長くなっても構わない（1トピックを表現するために必要な長さは確保する）。ただし複数トピックを詰め込むなら分割する。

## 入力タイプ別の補足

### LLMとのチャットから起こす場合

チャットには次の特徴があり、そのままアウトライン化すると質が低くなる。

- 話題が往復し、結論が後半に散らばる
- 試行錯誤の過程と結論が混在する
- ユーザーの質問とアシスタントの応答のラリー構造になっている

対処：

- ラリー構造を捨て、得られた知見を再構成する
- 最終的に到達した結論を最上位に置く
- 試行錯誤の過程はストーリーに必要なら残し、不要なら捨てる
- 「ユーザーがこう聞いた」「アシスタントがこう答えた」という形で書かない

### 論文・記事の散文から起こす場合

- 序論の問題提起・主要主張が最上位になることが多い
- 各セクションの結論が第2階層
- セクション内の議論が第3階層
- 数値・引用などのエビデンスが最下層

### メモ・断片から起こす場合

- まず断片同士をクラスタリング（似たトピックでまとめる）
- 各クラスタが「結局何を言いたいか」を命題として抽出
- クラスタ間の論理関係（因果、対比、構成要素など）を考えて階層化

---

## やらないこと

このスキルは構造抽出のためのもので、以下はやらない：

- **要約**：内容を短くすることが目的ではない。元のストーリーラインを骨格として残す。
- **創作・補完**：元のテキストにない主張を追加しない。元テキストから読み取れる範囲で構造化する。
- **批評**：元の論理に問題があっても、それを指摘するのは別タスク。アウトライン化では構造を可視化するに留める（ただし、ユーザーが指摘を求めた場合や、構造化の過程で論理ギャップが見つかったときに簡潔に注記するのは良い）。
