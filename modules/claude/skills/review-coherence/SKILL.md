対象の科学技術文章（論文・レポート・技術文書）の「一貫性」を専門的に校閲し、修正提案をファイルに保存せよ。

## 良い科学技術文章の基準

良い科学技術文章は以下を満たす：
- **シンプルで明解な論理**（大枠→詳細、つながりが自明、前提共有）
- **解釈ブレがない**（同一対象に同一用語、文法ミスなし）

## スコープ

このスキルは「論文全体として整合しているか」を担当する：
- 段落レベル、章レベル、文書全体レベルの構成
- ストーリーラインの一貫性
- 大枠→詳細の順序

※文レベルのつながりは review-clarity の担当

## 1. 深層分析

校閲を開始する前に、文章全体を通読して深く考えること。思考過程を `<thinking>` ブロックに明示せよ。

## 2. チェック項目

### 「大枠→詳細」の順序 ★重要★
- 各セクションは概要から始まり、詳細に進んでいるか
- 結論や主張を先に述べ、その後に根拠を示しているか
- 読者が全体像を把握してから細部に入れるか

```
NG: 詳細な手順 → 何を求めているか説明
OK: 目的の説明 → 詳細な手順
```

### 段落・章間のつながり ★重要★
- 段落と段落、セクションとセクションの接続が明確か
- 遷移文が適切に使われているか
- 読者が「なぜこのセクションが今出てくるのか」と感じないか

### 構造のシンプルさ ★重要★
- 議論が複雑に入り組んでいないか
- 一つの段落は一つの主題を扱っているか
- 脱線や横道がないか

### ストーリーラインの一貫性 ★重要★
- 文書全体を通して一貫したストーリーがあるか
- 各章が全体のストーリーに位置づけられているか
- 主張と結論が一貫しているか

### 論理の一貫性
- 前提と結論の整合性
- 主張の根拠が十分か
- 飛躍や論理的ギャップがないか
- 矛盾する記述がないか（特に Abstract-Conclusion 間）

### 主張の強さ
- 過度な一般化をしていないか
- データに基づいた主張か
- 限界（limitations）が適切に述べられているか

## 3. 論理的ギャップのパターン診断

| パターン | 説明 | 例 |
|---------|------|-----|
| 非順序 | A→B→Cの説明でBが抜けている | 条件設定の説明なく結果を提示 |
| 過度一般化 | 限定的データから広い結論 | "For all cases..." |
| 循環論法 | 結論を前提に使っている | 証明すべきことを仮定 |
| 相関と因果の混同 | 相関を因果として記述 | "A causes B"（相関のみ） |

## 4. 提出前の自己検証

以下をすべて確認してから保存すること：
- [ ] 対象文書を全て読んだ
- [ ] 全チェック項目を確認した
- [ ] 「十分な問題を見つけた」で止めず網羅的に探した
- [ ] 各問題に明確な場所と修正案がある

禁止：
1. 自己検証チェックリスト未完了で保存
2. 「問題なさそう」でセクションをスキップ
3. 場所と修正案のない曖昧な問題を報告

必須：
1. 問題がないセクションも明示的に「問題なし」と記載
2. 不確かな場合は「?」付きでフラグ

## 5. 修正提案の保存

すべての修正提案を `.claude/tmp/review-coherence.md` に書き込め。ファイルが存在しない場合は新規作成せよ。

各修正案は以下のフォーマットで記述すること：

```markdown
## [重要度] 問題の種類 - 場所

#### 要修正箇所
（英語で記述：該当箇所の原文）

#### 修正が必要な理由
（日本語で記述）

#### 修正提案
（英語で記述）

#### 対応
- [ ] 承認
- [ ] 拒否
- [ ] 条件付承認

##### modification
（承認後、ここに最終的な修正内容を記載）

---
```

重要度の分類：
- **Critical**: 文書の主張を損なう致命的な論理エラー、Abstract-Conclusion矛盾
- **Major**: 読者を混乱させる明確な論理の飛躍、構造問題
- **Minor**: より良い構成への改善提案

## 6. 完了報告

保存完了後、以下の形式でユーザーに報告せよ：

```
[Coherence] 校閲完了
Critical: X件 / Major: Y件 / Minor: Z件
保存先: .claude/tmp/review-coherence.md
```
